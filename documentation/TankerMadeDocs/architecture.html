Key Architectural Decisions

🌐 Blazor WASM over Server

  • Offline-first requirement: Server mode requires constant connection
  • Scalability: WASM scales better (computation on client)
  • Flexibility: Can add server later without rewriting UI

💾 SQLite over PostgreSQL Default

  • Simplicity: Single file, no server setup
  • Portability: Easy to backup (just copy file)
  • Sufficient: For single-user scenarios
  • Upgradeable: Can switch to PostgreSQL in production

🚀 Minimal APIs over Controllers

  • Simplicity: Less ceremony, faster to write
  • Modern: Recommended pattern in .NET 6+
  • Sufficient: Full REST API without boilerplate

🔐 JWT over Cookie Auth

  • WASM Compatibility: Cookies difficult with WASM + API separation
  • Stateless: No server-side session storage needed
  • Multi-domain: Works if frontend/backend on different domains

Module Interface

public interface IModule
{
    string Name { get; }
    string Version { get; }
    string Description { get; }

    // Lifecycle
    void RegisterServices(IServiceCollection services);
    void RegisterApiRoutes(IEndpointRouteBuilder routes);

    // UI Extensions
    Dictionary<string, Type> GetComponentExtensions();

    // Data Extensions (optional)
    void ConfigureDatabase(ModelBuilder modelBuilder);
}

Example: Crochet Module

public class CrochetModule : IModule
{
    public string Name => "Crochet";
    public string Version => "1.0.0";
    public string Description => "Crochet-specific features: hook sizes, stitch types, gauge tracking";

    public void RegisterServices(IServiceCollection services)
    {
        services.AddScoped<IStitchCalculatorService, StitchCalculatorService>();
    }

    public void RegisterApiRoutes(IEndpointRouteBuilder routes)
    {
        routes.MapGet("/api/crochet/hooks", GetHookSizes);
    }

    public Dictionary<string, Type> GetComponentExtensions()
    {
        return new Dictionary<string, Type>
        {
            ["ProjectDetail.AfterInfo"] = typeof(CrochetProjectInfo),
            ["PatternCreate.CustomFields"] = typeof(CrochetPatternFields)
        };
    }

    public void ConfigureDatabase(ModelBuilder modelBuilder)
    {
        // Optional: Add module-specific tables
        modelBuilder.Entity<CrochetProjectExtension>().ToTable("CrochetProjectExtensions");
    }
}

Security Architecture

Authentication Flow

  1. User enters credentials
  2. Client validates (offline) or calls API (server)
  3. JWT token stored in localStorage
  4. Token included in API requests
  5. Server validates JWT on each request

Authorization

  • Role-based: Viewer, Editor, Admin
  • Resource-based: Users can only access their own data
  • API-level: All endpoints require authentication
  • Client-side: UI elements hidden based on roles

Performance Considerations

Optimization Strategies

  • Lazy Loading: Routes and components loaded on demand
  • Code Splitting: Modules bundled separately
  • Caching: Reference data cached client-side
  • Pagination: Server-side for large datasets
  • Indexes: Database indexes on frequently queried fields
  • Asset Optimization: Compressed images and styles

⚠️ Known Limitations

  • IndexedDB Storage: ~50-100MB browser limit
  • WASM Bundle Size: Initial load time impact
  • Client-side Processing: Large datasets may be slow
  • Browser Support: Modern browsers required
