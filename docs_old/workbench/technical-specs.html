Technical Specifications

Implementation Details

Comprehensive technical specifications covering authentication flows, data synchronization strategies, API endpoints, and module system implementation for the TankerMade Workbench.

Authentication Flow

Offline Mode Authentication

1
User enters username/password on Login page
2
Client checks localStorage for user record
Simple hash comparison for password validation
3
If match, store offline-token (simple flag) in localStorage
4
Redirect to dashboard
All subsequent requests check for offline-token

Server Mode Authentication

1
User enters username/password on Login page
2
Client POSTs to /api/auth/login
Secure credential transmission over HTTPS
3
Server validates credentials, returns JWT
Token includes user ID, roles, and expiration
4
Client stores JWT in localStorage
Token automatically included in future requests
5
All subsequent API requests include JWT in Authorization header
Server validates JWT on each request

Data Synchronization Strategy

Offline → Server Sync

Process Flow

  1. User clicks "Sync to Server" in settings
  2. Export all local data to JSON
  3. POST to /api/sync/import with JSON payload
  4. Server imports with merge strategy:
    • Match by slug or unique identifier
    • If match found, use server version ("server wins")
    • If new, create on server
  5. Server returns sync report (created/updated/skipped counts)
  6. Client displays report

Server → Offline Sync

Process Flow

  1. User clicks "Sync from Server" in settings
  2. GET /api/sync/export (requires auth)
  3. Server exports user's data to JSON
  4. Client imports into IndexedDB with merge strategy:
    • Match by ID or slug
    • If match found, use server version ("server wins")
    • If new, create locally
  5. Client displays sync report

⚠️ Sync Strategy Notes

  • Server Wins: Simple conflict resolution for MVP
  • Export Before Sync: Manual backup recommended
  • Show Diff Preview: Preview changes before applying
  • Future Enhancement: Advanced merge strategies post-MVP

Example: Crochet Module Implementation

public class CrochetModule : IModule
{
    public string Name => "Crochet";
    public string Version => "1.0.0";
    public string Description => "Crochet-specific features: hook sizes, stitch types, gauge tracking";

    public void RegisterServices(IServiceCollection services)
    {
        services.AddScoped<IStitchCalculatorService, StitchCalculatorService>();
    }

    public void RegisterApiRoutes(IEndpointRouteBuilder routes)
    {
        routes.MapGet("/api/crochet/hooks", GetHookSizes);
        routes.MapPost("/api/crochet/gauge", CalculateGauge);
    }

    public Dictionary<string, Type> GetComponentExtensions()
    {
        return new Dictionary<string, Type>
        {
            ["ProjectDetail.AfterInfo"] = typeof(CrochetProjectInfo),
            ["PatternCreate.CustomFields"] = typeof(CrochetPatternFields),
            ["ProjectCreate.CustomFields"] = typeof(CrochetProjectFields)
        };
    }

    public void ConfigureDatabase(ModelBuilder modelBuilder)
    {
        // Optional: Add module-specific tables if needed
        modelBuilder.Entity<CrochetProjectExtension>().ToTable("CrochetProjectExtensions");
    }
}

Security Considerations

🔒 JWT Security

  • Token Storage: Consider HttpOnly cookies instead of localStorage to prevent XSS
  • Token Expiration: Short expiration (1 hour) with refresh mechanism
  • CSP Headers: Content Security Policy to prevent XSS attacks
  • HTTPS Only: All authentication over secure connections

🛡️ Data Protection

  • User Isolation: All queries filtered by UserId
  • Input Validation: Server-side validation for all inputs
  • SQL Injection Prevention: EF Core parameterized queries
  • File Upload Security: Validate file types and sizes

Caching Strategy

Client-Side Caching

  • Reference Data: Cache themes, colors, brands for 1 hour
  • User Stats: Cache for 5 minutes, invalidate on XP award
  • Achievement Definitions: Cache for 24 hours
  • User Profile: Cache until logout

Server-Side Caching

  • Achievement Definitions: In-memory cache, invalidate on admin changes
  • Reference Data: Redis cache if using multiple servers
  • User Statistics: Cache computed values, refresh on data change

Error Handling Strategy

Client-Side Error Handling

  • Global Error Boundary: Catch and display React/Blazor errors
  • Network Errors: Retry with exponential backoff
  • Validation Errors: Display field-specific errors
  • Offline Handling: Queue operations when offline

Server-Side Error Handling

  • Structured Logging: Log all errors with context
  • Exception Filters: Convert exceptions to standard responses
  • Validation: Return detailed validation errors
  • Rate Limiting: Prevent API abuse

⚠️ Known Technical Limitations

  • IndexedDB Browser Limits: ~50-100MB storage per origin
  • WASM Bundle Size: Initial load time increases with modules
  • Client-Side Processing: Large datasets may cause UI lag
  • Browser Compatibility: Modern browsers required (ES2020+)
  • Offline Search: Limited to client-side filtering
